## Context

The `TaskOutputModal.vue` component displays task execution output in a `<dialog>` element. It currently uses a fixed width of `36rem` and renders output as raw text in a `<pre>` block. Task output is generated by LLM-based task runners and frequently contains markdown formatting (headings, bullet lists, horizontal rules, code blocks) that would be more readable if rendered as HTML.

The frontend stack is Vue 3 + Vite + Tailwind CSS with no existing markdown rendering capability.

## Goals / Non-Goals

**Goals:**

- Make the output modal responsive, using approximately two-thirds of the viewport on wide screens
- Render task output as formatted markdown (headings, lists, separators, code blocks)
- Sanitize rendered HTML to prevent XSS from untrusted output content

**Non-Goals:**

- Editing task output (remains read-only)
- Syntax highlighting within code blocks (plain monospace is sufficient)
- Markdown rendering elsewhere in the app (scoped to the output modal only)

## Decisions

### Decision 1: Markdown library — `marked`

Use `marked` as the markdown-to-HTML parser.

- **Why**: Lightweight (~40KB), well-maintained, no dependencies, fast parsing. Used widely in the Vue ecosystem.
- **Alternatives considered**:
  - `markdown-it`: More extensible plugin system, but heavier and extensibility is not needed here.
  - `remark`/`rehype`: Full AST pipeline, significantly more complex for a simple render-to-HTML use case.
  - `showdown`: Less actively maintained than `marked`.

### Decision 2: HTML sanitization — `DOMPurify`

Use `DOMPurify` to sanitize the HTML output from `marked` before inserting into the DOM via `v-html`.

- **Why**: Task output comes from LLM execution and could contain arbitrary content. Using `v-html` without sanitization would be an XSS vector. DOMPurify is the standard choice for this, small (~15KB), and fast.
- **Alternatives considered**:
  - `marked` built-in sanitization: Deprecated and removed in v4+.
  - `sanitize-html`: Heavier, more configuration required.

### Decision 3: Modal width — responsive with max-width

Replace the fixed `w-[36rem]` with `w-[90vw] max-w-5xl` (Tailwind's `max-w-5xl` = `64rem`).

- On wide screens (>1280px): the modal will be `64rem` wide (~2/3 of a 1440px display).
- On narrower screens: the modal scales down to 90% of viewport width.
- This keeps the modal usable on all screen sizes without a breakpoint-based approach.

### Decision 4: Markdown styling — Tailwind prose

Apply Tailwind's `@tailwindcss/typography` plugin (`prose` class) to the rendered markdown container. This provides sensible default styling for headings, lists, code blocks, horizontal rules, etc. without writing custom CSS.

- The project already uses Tailwind, so this is a natural fit.
- The `prose` class can be scoped to just the output container.

## Risks / Trade-offs

- **[XSS from v-html]** → Mitigated by DOMPurify sanitization of all rendered HTML before DOM insertion.
- **[Output that isn't markdown]** → Plain text renders fine through `marked` — paragraphs and line breaks are preserved. No degradation for non-markdown output.
- **[New dependencies increase bundle size]** → `marked` (~40KB) + `DOMPurify` (~15KB) + `@tailwindcss/typography` (CSS only, tree-shaken) is a modest addition. These are well-established, stable packages.
